AWS CI/CD Environment Overview

Setting up team collaboration with Continuous Integration / Continuous Deployment can be easily accomplished using GitHub and AWS CodeBuild.

  • Continuous Integration automates building and testing a codebase changes prior to submitting those changes for approval.
  • Continuous Deployment automates deploying "stacks" into review or production environments.

GitHub, Feature Branch Workflow and AWS CodeBuild

To allow this guide to be immediately practical and easily implemented, we assumes you will use Git for version control, a GitHub account to host your Git repository, and a simple Feature Branch Workflow with GitHub Pull Requests to manage team collaboration. Focusing on this popular combination allows us to offer concrete and very specific usage instructions along with AWS CodeBuild projects to implement CI/CD. Other work flows and Git repository services may be used but each requires a different configuration.

The diagrams above show how the git commands and GitHub Pull Requests UI perform create/merge/delete on Git branches. The Branch Flow diagram show's the branches used in a simple Feature Branch Workflow. The local F1 branch is created off of the current main branch. A shared F1 branch is created to share the new feature in development. Next, the shared F1 branch is merged into the main branch and the main branch is tagged with a version number. And finally, both the local and shared F1 branches are deleted.

Here is a more detailed narrative for this workflow:

  1. Start Feature F1
    1. Developers solution folder is updated with latest content from main.
    2. Developer creates a local branch F1 off of remote main branch to work on a new feature.
  2. Local Dev on F1
    Each developer has their own AWS Account. They can use the AWS SAM CLI or the AWS Toolkit for Visual Studio (Windows) IDE to publish the stack to their own account. This allows rapid iteration and integration tests.
  3. Create PR for F1
    1. Developer edits code in solution folder.
    2. Developer publishes stack to their own account directly from their Solution Folder.
    3. Developer pushes F1 as a tracked branch (-u) to GitHub shared repository.
    4. Developer creates GitHub Pull Request. GitHub generates a PULL_REQUEST_CREATED event.
    5. AWS responds to event by running GitHubReviewCreateUpdate CodeBuild project:
      1. Branch F1 is checked out to CodeBuild Container
      2. dotnet build --Configuration Release # Build Solution
      3. dotnet test # Test Solution
      4. sam deploy ... # Create/Update the 'F1' AWS stack in Review Account
    6. Reviewer reviews F1 code and published F1 stack.
  4. Update PR for F1
    1. Developer updates code to incorporate suggestions from PR Reviewer(s).
    2. Developer publishes stack to their own account directly from their Solution Folder.
    3. Developer pushes F1 to GitHub shared repository. GitHub generates a PULL_REQUEST_UPDATED event.
    4. AWS responds to event by running GitHubReviewCreateUpdate CodeBuild project:
      1. Branch F1 is checked out to CodeBuild Container
      2. dotnet build --Configuration Release # Build Solution
      3. dotnet test # Test Solution
      4. sam deploy ... # Create/Update the 'F1' AWS stack in Review Account
    5. Reviewer review F1 code and published 'F1' stack.
  5. Merge PR for F1
    1. Delete the 'F1' AWS Stack in Review Account
    2. Update the 'Prod' AWS Stack in the Prod Account
  6. Finish Up F1
    1. Delete the local F1 branch
    2. Checkout the latest main branch
